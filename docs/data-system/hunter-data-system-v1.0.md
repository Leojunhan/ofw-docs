## 一、为什么先搭数据体系？

### 1.1 本质思考

**Dashboard不是UI问题，是数据问题。**

| 错误做法 | 正确做法 |
|:---------|:---------|
| 先画组件 → 再填数据 | 先定义数据 → 再决定呈现 |
| "这里放个卡片" | "这里要回答什么问题" |
| 追求视觉效果 | 追求信息决策价值 |

### 1.2 数据体系的三个层次

```
┌─────────────────────────────────────────────┐
│  第一层：数据模型                            │
│  （对象是什么、关系是什么）                   │
├─────────────────────────────────────────────┤
│  第二层：指标体系                            │
│  （健康/负载/效能/成本的具体指标）            │
├─────────────────────────────────────────────┤
│  第三层：数据呈现                            │
│  （Dashboard的优先级、聚合粒度、警戒阈值）    │
└─────────────────────────────────────────────┘
```

**先定义1和2，才能设计3。**

## 二、数据模型（第一层）

### 2.1 核心对象定义

Hunter系统的核心是**5个对象**：

| 对象 | 定义 | 生命周期 | 持久性 |
|:-----|:-----|:---------|:-------|
| **Agent** | 能力提供者，封装具体技能 | 注册→运行→离线→删除 | 持久 |
| **Skill** | Agent的具体能力单元 | 定义→绑定→调用→禁用 | 持久 |
| **Workflow** | 多Agent协同的编排逻辑 | 创建→发布→执行→下线 | 持久 |
| **Execution** | Workflow的一次执行实例 | 创建→执行→完成/失败→清理 | 临时→持久（记录） |
| **Resource** | Agent运行所需的资源（账号、话术等） | 注册→分配→消耗→评估 | 持久 |

### 2.2 对象关系图

```
┌──────────────┐
│    Agent     │
│  (能力载体)   │
└──────┬───────┘
       │
       │ 1:N
       ▼
┌──────────────┐     ┌──────────────┐
│    Skill     │────▶│   Workflow   │
│  (能力单元)   │ N:M │  (编排逻辑)   │
└──────────────┘     └──────┬───────┘
                            │
                            │ 1:N
                            ▼
                     ┌──────────────┐
                     │  Execution   │
                     │  (执行实例)   │
                     └──────┬───────┘
                            │
                            │ 回流
                            ▼
                     ┌──────────────┐
                     │   Agent      │
                     │   记忆更新    │
                     └──────────────┘
```

## 三、指标体系（第二层）

### 3.1 指标分层原则

Dashboard要回答的问题，按优先级分层：

| 层级 | 问题类型 | 用户关心程度 | Dashboard位置 |
|:-----|:---------|:-------------|:--------------|
| **P0 健康** | "系统活着吗？" | 第一眼看 | 顶部核心区 |
| **P1 负载** | "现在忙不忙？" | 第二眼看 | 核心区下方 |
| **P2 效能** | "干得怎么样？" | 深入查看 | 详情区 |
| **P3 成本** | "花了多少？" | 管理层看 | 汇报表 |

### 3.2 P0 健康指标

| 指标 | 定义 | 计算公式 | 数据源 | 警戒阈值 |
|:-----|:-----|:---------|:-------|:---------|
| **Agent在线率** | 在线Agent占比 | `online_count / total_count` | Agent表status字段 | <80% 报警 |
| **系统状态** | 整体健康判断 | 综合判断（在线率+错误率） | 多源聚合 | 指标任一异常则报警 |
| **核心Agent状态** | 关键Agent是否在线 | 指定Agent的status | Agent表 | 核心Agent离线立即报警 |

#### 数据聚合规则

| 粒度 | 实时性 | 用途 |
|:-----|:------:|:-----|
| 实时 | 秒级 | 健康监控 |
| 分钟聚合 | 1分钟 | 趋势图 |
| 小时聚合 | 1小时 | 历史对比 |

### 3.3 P1 负载指标

| 指标 | 定义 | 计算公式 | 数据源 | 警戒阈值 |
|:-----|:-----|:---------|:-------|:---------|
| **待处理任务数** | pending状态Execution数 | `count(status='pending')` | Execution表 | >50 预警 |
| **执行中任务数** | running状态Execution数 | `count(status='running')` | Execution表 | 接近Agent数 预警 |
| **队列等待时间** | 任务从创建到开始执行的平均时间 | `avg(start_time - created_time)` | Execution表 | >30秒 预警 |

#### 数据聚合规则

| 粒度 | 实时性 | 用途 |
|:-----|:------:|:-----|
| 实时 | 秒级 | 当前负载 |
| 分钟聚合 | 1分钟 | 负载趋势 |

### 3.4 P2 效能指标

| 指标 | 定义 | 计算公式 | 数据源 | 健康值 |
|:-----|:-----|:---------|:-------|:------:|
| **执行成功率** | 成功执行的占比 | `success_count / total_count` | Execution表 | >95% |
| **平均执行耗时** | 执行的平均时间 | `avg(duration_ms)` | Execution表 | <5000ms |
| **P95执行耗时** | 95%的执行耗时上限 | `percentile(duration_ms, 95)` | Execution表 | <10000ms |
| **Agent利用率** | busy状态Agent占比 | `busy_count / online_count` | Agent表 | 40-80% |
| **Skill利用率** | Skill被调用次数分布 | `usage_count排序` | Skill表 | 均衡 |

#### 数据聚合规则

| 粒度 | 实时性 | 用途 |
|:-----|:------:|:-----|
| 今日累计 | 天级 | 今日效能 |
| 7日趋势 | 天级聚合 | 周趋势 |
| 30日趋势 | 天级聚合 | 月趋势 |

### 3.5 P3 成本指标

| 指标 | 定义 | 计算公式 | 数据源 | 说明 |
|:-----|:-----|:---------|:-------|:-----|
| **今日执行数** | 今日执行总次数 | `count(created_at=today)` | Execution表 | 业务量 |
| **今日Token消耗** | （如果有LLM调用） | Agent上报 | Agent日志 | 成本追踪 |
| **API调用次数** | 外部API调用次数 | Execution步骤日志 | Execution表 | 成本追踪 |

#### 数据聚合规则

| 粒度 | 实时性 | 用途 |
|:-----|:------:|:-----|
| 今日累计 | 天级 | 今日成本 |
| 月度累计 | 月级 | 月度成本分析 |

│  │  P0 健康    │     │  P1 负载    │     │  P2 效能    │   │

## 四、数据流向（数据怎么流动）

### 4.1 数据产生源

| 数据源 | 产生频率 | 产生内容 |
|:-------|:---------|:---------|
| **Agent心跳** | 30秒/次 | status, last_heartbeat |
| **Execution执行** | 每次执行 | status, duration, result, error |
| **Skill调用** | 每次调用 | usage_count累加 |
| **Workflow执行** | 每次执行 | execution_count累加 |

### 4.2 数据流向图

```
┌──────────────┐
│   Agent      │
│  (心跳上报)   │
└──────┬───────┘
       │
       ▼ 30秒/次
┌──────────────┐     ┌──────────────┐
│  实时状态库   │────▶│  健康指标计算 │
│  (Redis)     │     │  (秒级聚合)   │
└──────────────┘     └──────┬───────┘
                            │
                            ▼ Dashboard P0区
                            
┌──────────────┐
│  Execution   │
│  (执行记录)   │
└──────┬───────┘
       │
       ▼ 每次执行
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  执行日志库   │────▶│ 分钟聚合     │────▶│ 天级聚合     │
│  (PostgreSQL)│     │ (Redis)      │     │ (PostgreSQL) │
└──────────────┘     └──────┬───────┘     └──────┬───────┘
                            │                    │
                            ▼ Dashboard P1区    ▼ Dashboard P2/P3区
```

### 4.3 数据聚合策略

| 层级 | 聚合粒度 | 存储位置 | 刷新频率 | 用途 |
|:-----|:---------|:---------|:---------|:-----|
| **L1 实时** | 不聚合 | Redis | 秒级 | P0健康指标 |
| **L2 分钟** | 1分钟聚合 | Redis | 1分钟 | P1负载趋势 |
| **L3 天级** | 1天聚合 | PostgreSQL | 1小时 | P2效能分析 |
| **L4 月级** | 1月聚合 | PostgreSQL | 1天 | P3成本报表 |

### 4.4 数据回流机制

```
Execution执行结果
       │
       ▼
┌──────────────────────────────────────┐
│              数据回流                  │
├──────────────────────────────────────┤
│                                      │
│  ① Skill.usage_count += 1            │
│                                      │
│ ② Workflow.execution_count += 1      │
│                                      │
│ ③ Agent记忆更新（如果有）             │
│     - 成功案例 → 记忆库               │
│     - 失败案例 → 异常模式库           │
│                                      │

│     - success_rate更新               │

```

## 五、Dashboard设计原则（第三层）

### 5.1 信息优先级原则

**用户第一眼应该看到什么？**

| 优先级 | 内容 | 视觉权重 |
|:-------|:-----|:---------|
| P0 | 系统健康状态 | 最大：顶部核心区，醒目颜色 |
| P1 | 当前负载 | 中等：核心区下方，清晰数字 |
| P2 | 效能趋势 | 较小：详情区，趋势图 |
| P3 | 成本报表 | 最小：折叠或单独页面 |

### 5.2 去冗余原则

**之前你提到的Dashboard问题：**

| 问题 | 原因 | 解决方案 |
|:-----|:-----|:---------|
| 信息过多，没有优先级 | 所有指标平铺 | 分层显示，P0>P1>P2>P3 |
| 快捷操作重复 | 快捷入口与侧边栏重复 | 去掉重复的快捷入口 |
| 导航冗余 | 多个入口指向同一页 | 合并导航，保持唯一入口 |

### 5.3 Apple设计思想映射

| Apple原则 | Dashboard应用 |
|:----------|:--------------|
| Less is more | P0指标优先，P3折叠隐藏 |
| 诚实 | 指标含义明确，不糊弄 |
| 一致性 | 所有卡片使用统一模板 |
| 留白 | 不填满屏幕，让眼睛休息 |

## 六、警戒阈值与告警规则

### 6.1 告警分级

| 级别 | 定义 | 触发条件 | 通知方式 |
|:-----|:-----|:---------|:---------|
| **紧急** | 系统不可用 | Agent在线率<50% | 钉钉+短信+邮件 |
| **严重** | 部分功能异常 | Agent在线率<80% | 钉钉+邮件 |
| **警告** | 性能下降 | 成功率<90% 或 P95>10s | 钉钉 |
| **提示** | 需关注 | 待处理>50 | Dashboard显示 |

### 6.2 告警抑制

| 规则 | 说明 |
|:-----|:-----|
| 同一告警5分钟内不重复发送 | 防止告警轰炸 |
| 告警恢复后发送恢复通知 | 让用户知道问题解决 |
| 夜间（22:00-08:00）仅紧急告警 | 尊重休息时间 |

## 七、总结

### 7.1 数据体系的核心价值

**数据体系是Dashboard的地基：**

| 层次 | 价值 |
|:-----|:-----|
| 数据模型 | 明确"有哪些数据、数据是什么" |
| 指标体系 | 明确"要回答什么问题、用什么指标" |
| 数据流向 | 明确"数据从哪来、怎么聚合、怎么呈现" |

### 7.2 Dashboard设计的下一步

有了数据体系，Dashboard设计变得清晰：

1. **P0区** → 健康指标（Agent在线率、系统状态）
2. **P1区** → 负载指标（待处理、执行中）
3. **P2区** → 效能指标（成功率、耗时）
