## 0. 执行摘要

### 0.1 产品定位

**FB 流量获取 Agent = 评论监控 + 智能互动 + 私信转化**

| 维度 | 定义 |
|------|------|
| **目标用户** | 菲律宾 OFW（有借贷需求的海外劳工） |
| **核心价值** | 低成本获取精准流量，自动化初步筛选 |
| **产品形态** | 后台 SaaS + Messenger 机器人 |
| **商业模式** | 内部使用（降低获客成本） |

---

### 0.2 核心目标

| 目标 | 指标 | 目标值 | 验证周期 |
|------|------|--------|---------|
| **获客效率** | 单账号日触达用户数 | 30-50 人/天 | 4 周 |
| **转化效率** | 触达→申请转化率 | >3% | 4 周 |
| **成本控制** | 单账号月成本 | <$200 | 持续 |
| **风险控制** | 账号月封号率 | <50% | 4 周 |
| **获客成本** | 单个申请成本 | <$2 | 4 周 |

---

### 0.3 关键决策

| 决策项 | 选择 | 原因 |
|--------|------|------|
| **技术路线** | 官方 API + 浏览器自动化混合 | 平衡风险与成本 |
| **账号策略** | 矩阵账号（5-10 个） | 接受封号现实，分散风险 |
| **AI 介入程度** | 话术生成 + 意图识别 | 核心交互 AI 化，操作人工审核 |
| **MVP 范围** | 评论监控 + 私信回复 | 先验证转化，再扩展功能 |

---

### 0.4 成功/失败标准（新增）

#### Phase 1 成功标准（4 周后评估）

**必须满足（全部）：**
- [ ] 获客成本 < $3/申请（放宽版）
- [ ] 转化率 > 2%（放宽版）
- [ ] 账号平均存活 > 10 天
- [ ] 无重大合规问题

**满足则进入 Phase 2，否则：**
- 获客成本 > $5 → 优化话术/渠道
- 转化率 < 1% → 重新设计对话流程
- 账号存活 < 5 天 → 降低频率/更换策略
- 重大合规问题 → 立即停止，咨询律师

#### 项目终止条件

满足以下任一条件，建议终止项目：
- 连续 4 周获客成本 > $5/申请
- 账号封号率 > 80%/月
- 收到律师函/监管警告
- 用户投诉率 > 5%

## 1. 产品概述

### 1.1 背景与问题

**业务背景：**
- OFW 信贷业务人工已跑通，但获客成本高（$4.5/人）
- FB 是 OFW 聚集地，但传统投放成本高、质量低
- 评论区有精准需求用户，但人工触达效率低

**核心问题：**
| 问题 | 现状 | 期望 |
|------|------|------|
| **获客成本高** | $4.5/申请 | <$2/申请 |
| **流量质量低** | 通过率 20-30% | 通过率 40-55% |
| **触达效率低** | 人工 50 条/天 | 自动 300 条/天 |
| **话术不统一** | 运营各自发挥 | 标准化 + 个性化 |

---

### 1.2 产品目标

**Phase 1（MVP，4 周）：**
- 实现评论监控 + 自动回复
- 实现 Messenger 自动对话
- 验证转化率>3%
- 验证封号率<50%/月

**Phase 2（扩展，8 周）：**
- 扩展至 10 个账号矩阵
- 优化话术，转化率提升至 5%
- 降低封号率至 30%/月

**Phase 3（规模化，12 周+）：**
- 扩展至 50+ 账号
- 获客成本降至$1-2/人
- 形成稳定获客渠道

---

### 1.3 用户画像

**C 端用户（OFW）：**

| 维度 | 描述 |
|------|------|
| **基本信息** | 25-45 岁，菲律宾籍，海外务工 |
| **收入水平** | 月收入$300-800（稳定汇款） |
| **借贷需求** | 医疗/教育/家用紧急支出 |
| **FB 使用习惯** | 日均 2-4 小时，活跃于 OFW 社群 |
| **语言偏好** | Taglish（Tagalog+English 混合） |
| **信任建立** | 需要多次互动，警惕诈骗 |

**B 端用户（运营人员）：**

| 维度 | 描述 |
|------|------|
| **职责** | 账号管理、话术审核、数据监控 |
| **使用频率** | 每日查看数据，每周优化话术 |
| **核心诉求** | 效率高、风险低、数据清晰 |

#### M2：评论监控

**用户故事：**
> 作为系统，我希望自动监控目标帖子的评论，提取有借贷意向的用户，以便运营人员触达。

**意图识别分类：**

| 意图类型 | 关键词示例 | 优先级 |

| **低意向** | "scam?", "legit?", 纯询问 | 人工审核 |

**对话流程设计：**

```
用户首次联系
    ↓
AI 欢迎语 + 产品介绍（1-2 句）
    ↓
询问基本信息（工作/收入/借款用途）
    ↓

    ↓
符合条件 → 发送申请链接
不符合 → 温和建议 + 留存
    ↓

    ↓
申请提交 → 转人工审批
```

## 5. 风险与应对

### 5.1 风险矩阵

| **数据泄露** | 低 (10%) | 致命 | 🔴 | 数据加密 + 访问控制 |

### 5.2 详细应对方案

#### 风险 1：FB 账号封禁

**应对策略：**
```
1. 账号矩阵：5-10 个账号，分散风险
2. 频率控制：单账号<30 次/天
3. 行为模拟：随机延迟、模拟人工操作
4. 快速检测：封号后 5 分钟内告警
5. 备用账号：随时可启用新账号
```

**监控指标：**
- 账号状态（每小时检测）
- 操作频率（实时统计）
- 封号率（每日统计）

---

#### 风险 2：用户举报

**应对策略：**
```
1. 话术优化：不推销、不骚扰、真诚回复
2. 频率控制：同一用户不重复触达
3. 举报监控：监控账号是否被举报
4. 快速响应：收到举报立即停止该账号操作
```

**话术红线：**
- ❌ 不承诺"一定通过"
- ❌ 不承诺"秒批"
- ❌ 不频繁发送
- ❌ 不骚扰用户

---

#### 风险 3：法律合规

**应对策略：**
```
1. 只抓取公开数据（不爬非公开信息）
2. 遵守菲律宾 Data Privacy Act 2012
3. 用户数据加密存储
4. 定期删除过期数据
5. 咨询律师，确保合规
```

#### 9.3.2 封号应对 SOP

```
发现封号 → 立即停止该账号所有操作 → 
检查进行中的对话 → 切换备用账号继续 → 
发送通知："Hi，刚才系统维护，现在继续为你服务~" → 
记录封号日志 → 分析封号原因 → 
7 天后尝试重新注册
```

**封号日志模板：**

| 字段 | 说明 |
|------|------|
| 封号时间 | YYYY-MM-DD HH:MM |
| 账号 ID | FB-A001 |
| 封号前操作 | 评论回复 28 条/天 |
| 可能原因 | 频率过高/用户举报/系统检测 |
| 影响对话数 | 12 个进行中对话 |
| 应对措施 | 切换备用账号，通知用户 |
| 后续计划 | 降低频率至 20 条/天 |
